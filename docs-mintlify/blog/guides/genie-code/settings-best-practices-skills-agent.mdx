## 4. Skills 생성 및 활용

### 4.1 Skills란?

Skills는 **재사용 가능한 작업 템플릿** 입니다. User Instructions가 "이렇게 해줘"라는 **규칙** 이라면, Skills는 "이 작업을 할 때는 이 절차를 따라라"라는 **레시피** 입니다. Agent 모드에서만 동작합니다.

**핵심 판단 기준**: 같은 유형의 작업을 여러 번 반복하면서 매번 동일한 절차와 규칙을 프롬프트에 작성하고 있다면, 그것은 스킬로 만들어야 합니다.

### 4.2 폴더 구조

**Workspace 스킬**: `/Workspace/.assistant/skills/{skill-name}/SKILL.md`

**User 스킬**: `/Workspace/Users/{email}/.assistant/skills/{skill-name}/SKILL.md`

### 4.3 SKILL.md 작성법

```markdown
---
name: data-quality-check
description: 테이블의 데이터 품질을 검증하고 리포트를 생성합니다
---

# 데이터 품질 검증 스킬

## 이 스킬을 사용하는 경우
- 사용자가 테이블의 데이터 품질을 확인하고 싶을 때

## 실행 절차
### 1단계: 테이블 프로파일링
### 2단계: 품질 규칙 점검
### 3단계: 리포트 생성
```

### 4.5 Workspace 스킬 vs User 스킬 선택 기준

| 판단 기준 | Workspace 스킬 | User 스킬 |
|-----------|---------------|-----------|
| **사용 범위** | 팀 전체가 동일한 절차를 따라야 할 때 | 개인 업무에 특화된 반복 작업 |
| **표준화** | 파이프라인 템플릿, 배포 절차 | 개인 분석 패턴, 실험 워크플로 |
| **관리** | 관리자가 버전 관리 | 개인이 자유롭게 수정 |

> **팁**
처음에는 User 스킬로 만들어서 검증한 뒤, 팀에 유용하면 Workspace 스킬로 승격시키세요.

## 6. Agent 모드 실전 팁

### 6.1 컨텍스트 격리 전략

**원칙: 하나의 세션 = 하나의 작업 단위**

| 상황 | 권장 행동 |
|------|----------|
| 새로운 데이터셋을 분석할 때 | 새 채팅 세션 시작 |
| 이전 분석 결과를 기반으로 후속 작업할 때 | 같은 세션에서 계속 |
| 에러가 반복되어 세션이 혼란스러울 때 | 새 채팅 세션에서 문제를 재정의 |

> **주의**
브라우저 탭을 전환하면 Agent 모드가 **일시정지** 됩니다.


### 6.2 권한 관리: "Always Allow" 주의

**사용해도 되는 경우**: 읽기 전용 작업, sandbox/dev 환경

**사용하면 안 되는 경우**: 프로덕션 데이터 쓰기, 외부 시스템 전송, 파괴적 작업

### 6.3 작업 정의 방식: 목표 중심 + 제한 설정

**효과적인 프롬프트:**

```
@main_catalog.sales.orders 테이블에서 상위 10 고객의 매출 분석 대시보드를 만들어줘.

제한사항:
- 최근 90일 데이터만 사용
- 개발 중에는 1000행으로 제한하여 먼저 테스트
- 결과는 dev_catalog.sandbox.simyung에 저장
- Plotly 차트 사용
```

### 6.4 보안 제한 설정

```
분석 진행하되 다음 규칙을 반드시 지켜줘:
- DROP TABLE, DELETE, UPDATE, ALTER 문을 실행하지 마
- 프로덕션 카탈로그(prod_catalog)에 절대 쓰기하지 마
- 외부 API 호출 시 민감 데이터를 전송하지 마
```

> **팁**
보안 규칙은 User Instructions에 한 번 설정해두면 매 프롬프트에 반복하지 않아도 됩니다. 하지만 민감한 작업에서는 **프롬프트에서 한 번 더 강조** 하는 이중 방어가 안전합니다.


### 6.6 무한 루프 대응

같은 에러가 2-3번 반복되면 즉시 **Stop** → 에러 메시지 복사 → **새 채팅 세션** 에서 재정의

> **주의**
같은 세션에서 계속 시도하지 마세요. 세션의 컨텍스트가 에러 정보로 가득 차 있어서 벗어나기 어렵습니다.

## 7. 추천 설정 조합 (역할별)

| 역할 | User Instructions 중점 | 추천 Workspace 스킬 | 추천 MCP 서버 |
|------|----------------------|-------------------|-------------|
| **데이터 엔지니어** | PySpark 패턴, DLT 규칙 | 파이프라인 템플릿, 품질 검증 | Databricks, GitHub, Slack |
| **데이터 분석가** | SQL 스타일, 시각화 선호 | 대시보드 생성, 리포트 자동화 | Databricks, Slack |
| **ML 엔지니어** | MLflow 규칙, 모델 문서화 | 모델 배포, 피처 엔지니어링 | Databricks, GitHub, Slack |
| **플랫폼 관리자** | 거버넌스 규칙 | 권한 감사, 비용 분석 | Databricks, JIRA, Slack |

### 설정 우선순위 가이드

1. **Workspace Instructions 확인** — 관리자가 이미 설정했는지 확인
2. **User Instructions 작성** — 역할, 선호 라이브러리, 보안 규칙 정의
3. **MCP 서버 추가** — Databricks MCP부터 시작
4. **Skills 생성** — 반복 작업이 2-3번 이상 발생하면 스킬로 전환

> **참고**
설정은 **살아있는 문서** 입니다. "이걸 매번 말하고 있네"가 User Instructions 추가 타이밍이고, "이 절차를 또 설명하고 있네"가 스킬 생성 타이밍입니다.
