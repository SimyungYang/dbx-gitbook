Genie Code의 설정 화면 구성, User Instructions 작성법, Workspace Instructions 설정을 다룹니다.

> **최종 업데이트**: 2026-04-08

Genie Code를 처음 열면 프롬프트 입력만 보이지만, 설정을 제대로 구성하면 **동일한 프롬프트에서도 완전히 다른 수준의 결과** 를 얻을 수 있습니다.

---

## 1. 설정 화면 구성 요소

| 설정 항목 | 역할 | 영향 범위 | 누가 설정하는가 |
|-----------|------|-----------|----------------|
| **MCP 서버** | 외부 도구 연동 | 해당 사용자의 Agent 모드 세션 | 개인 사용자 |
| **패널 뷰** | Docked/Side 전환 | UI 레이아웃만 변경 | 개인 사용자 |
| **User Instructions** | 개인용 지속적 지침 | 해당 사용자의 모든 세션 | 개인 사용자 |
| **Workspace Instructions** | 조직 전체 지침 | 워크스페이스 전체 | 워크스페이스 관리자 |
| **Skills** | 재사용 가능한 작업 템플릿 (Agent 모드 전용) | User/Workspace 스킬 | 개인 또는 관리자 |

> **참고**
Genie Code는 응답 생성 시 **Workspace Instructions를 User Instructions보다 우선** 적용합니다.

## 2. User Instructions 작성법

### 2.1 User Instructions란?

User Instructions는 Genie Code에게 **"나는 이런 사람이고, 이렇게 일해줬으면 좋겠어"** 라고 지속적으로 알려주는 개인화 지침입니다.

### 2.2 생성 방법

**방법 1: UI에서 직접 설정** — Settings > User instructions > Add instruction file

**방법 2: 파일 직접 생성** — `/Workspace/Users/{your-email}/.assistant_instructions.md`

> **주의**
최대 길이는 **20,000자** 입니다. 가장 중요한 지침을 상위에 배치하세요.


### 2.4 좋은 User Instructions 예시

```markdown
## 내 역할과 컨텍스트
- 나는 데이터 엔지니어이며, 주로 Delta Lake 파이프라인과 SQL 분석 작업을 합니다
- 프로덕션 카탈로그는 `main_catalog`, 개발용은 `dev_catalog`

## 코딩 스타일
- PEP 8, SQL 대문자 키워드, snake_case, docstring 필수, 타입 힌트

## 선호 라이브러리
- 데이터 처리: PySpark / 시각화: Plotly / ML: MLflow + scikit-learn

## 보안 규칙 (절대 위반 금지)
- main_catalog.prod.* 절대 수정 금지
- DROP TABLE, DELETE, UPDATE 문 생성 금지
- 임시 결과는 dev_catalog.sandbox.simyung에만 저장
- 민감 컬럼(ssn, email, phone) 마스킹 처리

## 응답 스타일
- 코드 블록 앞에 간단한 설명, 한국어로 응답
```

### 2.5 나쁜 User Instructions와 개선 방향

- **너무 모호한 지침**: "좋은 코드를 작성해주세요" → 구체적 기준 정의 필요
- **상충되는 지침**: 간결함과 상세한 에러 처리/문서화는 본질적으로 상충 → 우선순위 설정
- **역할과 무관한 과잉 지침**: JavaScript, CSS, DevOps 규칙 → Databricks 환경에 맞는 지침만

> **팁**
User Instructions는 **자주 반복하게 되는 지침** 만 포함하세요. 일회성 요구사항은 프롬프트에서 직접 지시하는 것이 더 효과적입니다.

## 3. Workspace Instructions 설정

### 3.1 왜 Workspace Instructions가 필요한가?

User Instructions가 "나는 이렇게 해줘"라면, Workspace Instructions는 **"우리 조직에서는 반드시 이렇게 해야 해"** 입니다.

### 3.2 생성 방법

워크스페이스 관리자가 다음 경로에 파일을 생성합니다:
```
/Workspace/.assistant_workspace_instructions.md
```

> **주의**
**워크스페이스 관리자만** 생성하고 수정할 수 있습니다.


### 3.3 효과적인 Workspace Instructions 예시

```markdown
## 조직 코딩 표준
- 모든 테이블 참조는 Unity Catalog 3-level 네이밍 (catalog.schema.table)
- 파이프라인 코드에 반드시 try-except 에러 핸들링 포함

## 데이터 레이어 네이밍 컨벤션
- Bronze: raw_{소스시스템}_{엔티티}
- Silver: clean_{도메인}_{엔티티}
- Gold: agg_{비즈니스영역}_{메트릭}

## 데이터 접근 정책
- PII 테이블: hr.employees, crm.contacts, crm.customers → mask_pii() 필수
- 프로덕션 카탈로그(prod_catalog) DDL/DML 생성 금지

## SQL 스타일 가이드
- 키워드 대문자, 의미 있는 alias, CTE 우선, 개발 시 LIMIT 1000

## 보안 필수 규칙
- 자격 증명을 코드에 절대 포함 금지 → dbutils.secrets 사용
```

### 3.4 User Instructions와 Workspace Instructions의 역할 분리

| 구분 | User Instructions에 적합 | Workspace Instructions에 적합 |
|------|------------------------|------------------------------|
| **코딩 스타일** | 개인 선호 | 조직 표준 (네이밍 컨벤션 등) |
| **데이터 접근** | 자주 쓰는 스키마/테이블 | PII 정책, 프로덕션 접근 제한 |
| **보안** | 개인 sandbox 스키마 | 조직 전체 보안 규칙 |
