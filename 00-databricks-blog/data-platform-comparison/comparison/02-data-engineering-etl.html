데이터 엔지니어링 / ETL

Lakeflow(DLT)의 선언적 파이프라인이 복잡한 ETL을 단순화합니다

핵심: 선언적(Declarative) vs 명령적(Imperative) ETL

전통적 ETL은 "어떻게" 데이터를 처리할지 모든 단계를 코드로 작성해야 합니다. Databricks Lakeflow(DLT)는 "무엇을" 원하는지만 선언하면 실행 계획, 오류 복구, 데이터 품질 검증, 의존성 관리를 플랫폼이 자동 처리합니다. 배치와 스트리밍도 동일한 코드로 통합됩니다.

Databricks Lakeflow 파이프라인 흐름

Raw Data (S3/ADLS/GCS) → Lakeflow Ingestion (Auto Loader) → DLT: Bronze (Raw) → DLT: Silver (Cleaned + Expectations) → DLT: Gold (Aggregated) → SQL Analytics / ML / Apps

배치와 스트리밍이 동일한 선언적 코드로 동작 | Expectations로 품질 자동 검증 | Lineage 자동 추적

Databricks Lakeflow가 선택받는 이유

선언적 파이프라인

"무엇을" 정의하면 "어떻게"는 플랫폼이 자동 처리. 오류 복구, 재시도, 의존성 순서, 리소스 관리까지 모두 자동화됩니다.

배치-스트리밍 통합

동일한 DLT 코드로 배치와 스트리밍을 전환합니다. 별도의 스트리밍 인프라 없이 실시간 파이프라인을 구축할 수 있습니다.

APPLY CHANGES (CDC)

코드 한 줄로 SCD Type 1/2 CDC를 구현합니다. DMS→Glue→Redshift 같은 다단계 파이프라인이 불필요합니다.

DLT Expectations

데이터 품질 규칙을 파이프라인 정의에 선언합니다. 위반 시 FAIL/DROP/WARN을 선택하고, 메트릭이 자동 수집됩니다.

Auto Loader

클라우드 스토리지의 새 파일을 자동 감지하고 증분 처리합니다. 스키마 진화도 자동으로 처리하여 수동 개입이 불필요합니다.

Genie Code로 파이프라인 생성

자연어로 "매일 S3에서 고객 데이터를 가져와 정제해줘"라고 말하면 DLT 파이프라인 코드를 자동 생성합니다.

결론: 데이터 엔지니어링 / ETL

다른 플랫폼은 배치, 스트리밍, CDC, 오케스트레이션을 별도의 서비스로 조합해야 합니다. Databricks는 Lakeflow 하나로 모든 것을 통합하며, 선언적 모델로 복잡성을 근본적으로 제거합니다.
